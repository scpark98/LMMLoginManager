## 1. 패스워드 마스크 글리프(●) 의 거친 외곽 — AA 누락

### 문제
패스워드 edit (`m_edit_pw`) 의 마스크 문자 ● (U+25CF BLACK CIRCLE) 가 매끈한 원이 아닌 거친 모양으로 표시됨. 확대해보니 각 원의 좌우 가장자리에 빨강/파랑 fringe 와 톱니가 명확.

### 원인
- `CSCStaticEdit::OnPaint` 안에서 텍스트는 GDI+ `DrawString` 이 아닌 **GDI `dc.TextOut()`** 으로 그려진다.
- GDI TextOut 의 안티앨리어싱은 `HFONT::lfQuality` 가 결정한다.
- 이전에 한 줄 분기로 추가했던 `g.SetTextRenderingHint(m_password ? AntiAlias : ClearType)` 는 **GDI+ 경로용** 이라 실제 렌더링에 0 영향 — 죽은 코드였다.
- ● 는 원형 외곽선이라 ClearType 의 grid hinting 효과가 없고, grayscale AA 가 없으면 톱니와 subpixel fringe 가 그대로 노출된다.

### 수정 방향
- `rebuild_font()` 가 `m_password` 를 보고 lfQuality 를 자동 결정 — 패스워드 모드면 로컬 LOGFONT 사본의 `lfQuality = ANTIALIASED_QUALITY` 로 강제 후 `CreateFontIndirect`.
- `m_lf` 원본은 건드리지 않음 → 외부에서 `set_font_antialias` 로 명시 설정한 값이 평문 모드에서 그대로 보존됨.
- `m_password` 가 변경되는 두 지점에서 `rebuild_font()` 호출 추가:
  - `set_password_mode(...)` — 모드 진입/이탈 시
  - `action_password_toggle` — UI 토글 시
- 죽은 코드(GDI+ hint 분기) 는 단일 `ClearTypeGridFit` 호출로 복원.

### 이전 vs 이후
| | 이전 | 이후 |
|---|---|---|
| 평문 (자모) | ClearTypeGridFit → 또렷 | 동일 |
| 패스워드 (●) | ClearTypeGridFit → 톱니 + R/B fringe | **ANTIALIASED_QUALITY → 매끈한 회색 그라데이션** |

## 2. `service_command` 정공법 재작성 (Common/Functions.cpp)

### 문제 (다수)
1. `OpenService` 에 SCM 용 access mask `SC_MANAGER_ALL_ACCESS` 를 전달 — 서비스용 `SERVICE_*` 가 맞다. 비트 배치가 우연히 일부 일치해 "동작은 했지만" `restart` 의 `SERVICE_START` 비트가 빠지는 등 잠재 결함.
2. STOP 완료 polling 이 `while (state != STOPPED) QueryServiceStatus(...)` — **무한 루프 + Sleep 없음 + 타임아웃 없음**. QueryServiceStatus 가 실패하면 영영 hang + CPU 100% spin.
3. `QueryServiceStatus` 실패 시 `status.dwCurrentState = 0` 인 채로 STOP 분기 진입 → 이미 stopped 인 서비스에 STOP 호출 → `ERROR_SERVICE_NOT_ACTIVE`.
4. **`DeleteService` 후 marked-for-delete 풀림 미검증** — DeleteService 는 "삭제 요청" 만 큐잉. 그 사이 동일 이름 install 은 `ERROR_SERVICE_MARKED_FOR_DELETE` 로 실패. **사용자가 본 "한두 번은 잘 되다 갑자기 ERROR_SERVICE_DOES_NOT_EXIST" 의 핵심 원인.**
5. delete 성공 시 `status_code = SERVICE_STOPPED` 로 의미 변형 — 호출자가 상태로 해석 못 함.
6. `install` / `start` cmd 미지원 → 호출자가 `ShellExecute("LMMAgent.exe -install")` 비동기 우회.
7. `error_code` 가 성공/실패 경로마다 일관되지 않게 설정.

### 수정 방향
- **cmd 별 최소 권한 mask**: `query=SERVICE_QUERY_STATUS`, `start=SERVICE_START|QUERY_STATUS`, `stop=SERVICE_STOP|QUERY_STATUS`, `delete=DELETE|SERVICE_STOP|QUERY_STATUS`.
- **MSDN 표준 폴링 헬퍼 `sc_wait_for_state`** — `QueryServiceStatusEx` + `dwWaitHint`/`dwCheckPoint` 추적, Sleep 클램프 (100ms~10s), 전체 30s wall-clock 타임아웃, checkpoint 정체 감지.
- **DeleteService 후 동기 폴링** — hService close → `OpenService` 가 `ERROR_SERVICE_DOES_NOT_EXIST` 반환할 때까지 폴링 (전체 30s). 실제 SCM 에서 사라짐을 보장한 뒤 반환.
- `start` 분기 추가 + `service_install` 별도 함수로 분리 (CreateService 동기 호출, marked-for-delete 만나면 자동 재시도).
- **`error_code` 단일 계약**: 0 = 성공, 그 외 = WinAPI 에러. delete 성공 시 0 반환 (서비스 없음의 정확한 의미).
- 멱등성: `stop` 시 서비스 없음 → `SERVICE_STOPPED` 반환 (이미 멈춰있음과 동의).

### 이전 vs 이후
| | 이전 | 이후 |
|---|---|---|
| 안정성 | "한두 번은 잘 되다 갑자기 실패" race | marked-for-delete 동기 보장으로 race 제거 |
| 무한 hang | STOP polling 무한 루프 가능 | 30s 타임아웃 + Sleep 클램프 |
| CPU 사용 | spin 100% | dwWaitHint/10 기반 정상 polling |
| install 자체 | `ShellExecute` 우회만 | `service_install` 동기 호출 가능 (선택지) |

## 3. LMMLoginManager 의 `service_stop()` / `service_start()` 정공법

### 문제
- `service_start()` 에 인위적 `Wait(1000)` 두 개 (service_stop 후 / `ShellExecute("-install")` 후) — **직전 명령의 성공을 보장하지 못한다.**
- `while (true) { query → status<1 → Sleep(1000) → continue }` — **종료 조건 없음**. install 영구 실패 시 UI 영영 hang.
- 폴링 안의 `logWrite` 가 포맷 specifier 없는데 인자 3개 — 포맷/인자 불일치.
- `service_stop()` 가 옛 `service_command("delete")` 의 "status==1=성공" 의미에 의존 — 새 함수의 "status==0=성공" 계약과 충돌. delete 성공인데 "fail" 로그 + `return false` (동작은 service_start 가 반환값 무시해서 무해하지만 로그가 잘못 찍힘).
- `service_stop()` 의 query 가 0 반환 + error_code 가 `ERROR_SERVICE_DOES_NOT_EXIST` 가 아닌 경우 (ACCESS_DENIED 등) 그냥 `return true` — 서비스가 있는데 모르고 진행할 위험.
- `ShellExecute("-start")` 후 완료 검증 없음 — start 실패해도 LoginManager 가 알 수 없음.

### 수정 방향
- **인위적 Wait 모두 제거** — polling 이 정답.
- **install polling**: `GetTickCount` 기반 wall-clock 타임아웃 30s, 폴링 500ms, 에러 종류별 분기 (`ERROR_SERVICE_DOES_NOT_EXIST` 만 폴링 지속, 다른 에러는 즉시 false). 타임아웃 시 명확한 로그 + false 반환.
- **`-start` 도 동일 패턴** — `SERVICE_RUNNING` 까지 30s 폴링. `STOPPED`/`START_PENDING` 은 폴링 계속, 그 외 에러는 즉시 false. STOPPED 지속 시 타임아웃이 LMMAgent `-start` 호출 실패를 잡음.
- **`service_stop()` 재작성**: error_code 기반 판별 (0 = 성공). query 실패 시 정확히 분기 (ERROR_SERVICE_DOES_NOT_EXIST → 멱등 성공, 그 외 → false 반환).
- logWrite 의 포맷/인자 미스매치 정정.

### 이전 vs 이후
| | 이전 | 이후 |
|---|---|---|
| 인위적 Wait | 2초 (보장 없음) | 0 — polling 이 동기 보장 |
| install 실패 | 영영 hang | 30s 타임아웃 + false 반환 + 로그 |
| start 검증 | 없음 (성공/실패 알 수 없음) | SERVICE_RUNNING 까지 polling |
| service_stop 정확성 | delete 성공인데 "fail" 로그 | error_code 기반 정확한 성공/실패 판별 |

## 4. 자동 로그인 시 잘못된 PW 가 정상 로그인으로 처리되는 버그 (보안/UX 결함)

### 문제
- "부팅 시 자동 로그인" 옵션 체크 + 잘못된 암호 입력 → "로그인 실패" 가 아닌 **정상 로그인 처리**.
- 옵션 미체크 시에는 정상적으로 "암호 틀림" 표시.

### 원인 추적
LMMHost 의 `ConnectionThread::SetAgentIdPwToken` (라인 99-102):
```cpp
if (agentToken != _T(""))
    m_isAutoLogin = TRUE;
```
**token 이 비어있지 않으면 무조건 자동 로그인 모드** → `KmsConnect` 가 `NMS_CS_AUTO_LOGIN` 송신 → 서버는 **token 만 검증, password 무시** → 잘못된 PW 도 LOGIN_OK 응답.

현재 LoginManager 의 `OnBnClickedButtonLogin` LOGIN_BEFORE 분기가 사용자 수동 로그인 시 token 을 비우는 처리를 누락 → 이전 정상 로그인 때 받은 유효 토큰이 ini 에 그대로 남아 → LMMHost 가 자동 로그인 모드로 처리.

renewal 이전 `LoginDlg.cpp:101-119` 에는 다음 처리가 있었으나 renewal 과정에서 누락:
1. `Config::SaveAgentToken("")` — 토큰 비우기
2. `Config::SaveManualLoginStatus(1)` — 수동 로그인 표시
3. `set_registry_int(... "is auto setup", 0)` — AutoPatcher 흐름 리셋
4. `set_registry_int(... "auto setup installed", 0)`

### 수정 방향
LMMHost 무손상 원칙대로 **LMMLoginManager 측에서 누락된 처리 복원**:
- `OnBnClickedButtonLogin` LOGIN_BEFORE 분기에 위 4가지 추가.
- 추가: `UdpSocket.cpp` 의 `MANUAL_LOGIN_STATUS = 0` 리셋이 `AfxGetApp()->WriteProfileInt` (레지스트리) 로 잘못 쓰고 있었음. LMMHost 는 `GetPrivateProfileInt` 로 ini 에서 읽음 → 위치 불일치. `theApp.m_ini[...]` 로 정정 (6곳).

### 이전 vs 이후
| 시나리오 | 이전 | 이후 |
|---|---|---|
| 옵션 체크 + 정상 PW | LOGIN_OK ✓ | LOGIN_OK ✓ |
| **옵션 체크 + 잘못된 PW** | **LOGIN_OK (버그)** | **"암호 틀림" 표시 ✓** |
| 옵션 미체크 + 잘못된 PW | "암호 틀림" ✓ | "암호 틀림" ✓ |
| 수동 로그인 실패 후 옵션 상태 | 자동 해제 (의도 어긋남) | 유지 (manualLoginStatus=1 보존) |
| MANUAL_LOGIN_STATUS 동기화 | 레지스트리 ↔ ini 불일치 | ini 일관 동기 |

## 5. `service_start()` fast-path — 정상 고객의 매번 5~10초 → ~즉시

### 문제
LoginManager 가 로그인 버튼을 누를 때마다 `delete + install + start` 순서로 진행 — **매번 5~10초 소요**.

배경: 우리 회사 LMM 제품을 파생해 다른 회사에 솔루션으로 납품하는 경우, 두 제품이 **같은 서비스 이름**을 사용. 파생 제품의 서비스가 등록된 상태에서 정식 SAS 제품을 구동하면 잘못된 LMMAgent.exe 가 실행되어 에러 → 안전을 위해 매번 reinstall.

하지만 **정상 고객은 한 제품만 사용** → 이 안전 마진이 매번 5~10초의 비용.

### 수정 방향
**조건부 분기 (binary path 비교)**:
- Common 에 `service_get_binary_path(name, out_path, error_code, detail*)` 신규 — `QueryServiceConfig::lpBinaryPathName` 반환.
- LoginManager 의 `service_start()` 진입 시:
  1. query → 서비스 없음 → 기존 full path (install + start)
  2. query → 서비스 있음 + binary == 우리 경로 + `SERVICE_RUNNING` → **즉시 return true**
  3. query → 서비스 있음 + binary == 우리 경로 + 그 외 상태 → install 스킵, **start 만 + polling**
  4. query → 서비스 있음 + binary != 우리 경로 (파생/충돌) → 기존 full path
- binary path 비교는 정규화 (`paths_equal`): 대소문자 무시 + 슬래시 통일 + trim.
- LMMHost 의 `installService` 가 `SERVICE_AUTO_START` 로 등록하므로 **재부팅 후 자동 시작된 RUNNING 상태가 정상** → 정상 고객 두 번째 로그인부터 거의 모두 케이스 2.

### 이전 vs 이후
| 환경 | 이전 | 이후 |
|---|---|---|
| **정상 고객, 재부팅 후 (RUNNING)** | ~5~10초 | **~즉시 (0.1초)** |
| **정상 고객, 서비스 STOPPED** | ~5~10초 | **~1~2초** |
| 첫 로그인 (서비스 없음) | ~5~10초 | ~5~10초 (동일) |
| 개발/파생 충돌 (binary 다름) | ~5~10초 | ~5~10초 (안전 보장) |

## 6. 로그 정리 (운영 noise 제거)

진단 가치는 낮고 운영 환경에서 매 로그인/폴링마다 출력되는 verbose 로그 제거:
- fast-path RUNNING/STOPPED 진입 시 정상 흐름 로그 (2개)
- install polling 의 `waiting for the service to be installed...` (500ms 마다)
- start polling 의 `waiting for the service to start...` (500ms 마다)

**유지된 로그**: 충돌 진단(binary 다름), 모든 실패/타임아웃, install/start 시작·성공 마무리, service_stop 의 성공/실패 진단. 디버깅 시 충분.

## 7. 잔재 LMMAgent.exe / 자식 프로세스 강제 정리

### 문제
- 로그인 실패를 반복하면 사용자 세션에 `LMMAgent.exe -controlservice -slave` (그리고 경우에 따라 `-desktopserver`) 가 살아남는다. 시간 지나면 알아서 사라지기도 하지만 계속 남는 경우도 있음 → 누적되며 시스템 부담.
- 정공법: 로그인 실패/로그아웃 등 서비스가 정상 실행 중이 아닌 모든 상태에서 LMMAgent.exe 인스턴스가 살아있을 수 없어야 한다.
- 단순 `taskkill /f /im LMMAgent.exe` 는 금지 — **LMM 3.0 SE** 같은 다른 제품도 동일 프로세스 이름을 사용하므로 동명 다른 제품을 같이 죽인다.

### 원인 추적

#### A. `service_stop()` 가 SCM 의 `-service` 만 정지
- SCM 의 STOP 시그널은 `LMMAgent.exe -service` (SYSTEM, session 0) 에만 전달된다.
- `-service` 가 user 세션에 별도 띄운 `-controlservice -slave` / `-desktopserver` 는 SCM 과 무관한 별도 프로세스. STOP 시그널 못 받음 → 살아남는다.
- LMMHost 가 `-service` 종료 시 자식들 정리하는 로직이 모든 경로에 일관되게 동작하지 않는 듯.

#### B. `kill_process_by_fullpath` (Common) 자체의 정확성 결함
- **첫 매칭 1개만 종료 후 break** — 같은 fullpath 의 여러 인스턴스가 떠있어도 한 번 호출에 1개만 종료.
- `OpenProcess` 결과 NULL 검증 없음 → NULL handle 에 `TerminateProcess`/`CloseHandle` 호출.
- `CreateToolhelp32Snapshot` 의 `INVALID_HANDLE_VALUE` 검증 없음.
- 권한 mask 가 `PROCESS_ALL_ACCESS` (광범위) — 권한 부족으로 false negative 증가.
- 함수 시작에서 `get_process_running_count` 호출 후 본문에서 같은 snapshot 다시 만들어 중복.

#### C. `terminate_other_processes()` 의 `taskkill /f /im name.exe` 부작용
- 같은 이름의 *모든* 프로세스를 죽인다 (경로 무관).
- LMM 3.0 SE 등 다른 제품의 nScreenClient.exe / nFTDClient.exe / nSSDClient.exe / hwmon.exe 도 종료시키는 부작용.
- `ShellExecute` 비동기 — 함수 반환 시점에 종료 보장 X.

### 수정 방향

#### A. Common 의 `kill_process_by_fullpath` 정정
- **`break` 제거** — 매칭되는 모든 인스턴스 종료.
- snapshot / OpenProcess 결과 NULL 검증 추가.
- 권한 분리: 조회용 `PROCESS_QUERY_INFORMATION|PROCESS_VM_READ` + 종료용 `PROCESS_TERMINATE` 별도 OpenProcess.
- `get_process_running_count` 호출 제거 (중복 enumerate).
- 반환값 의미 명확화: `>0` = 종료된 개수, `0` = 매칭 없음 또는 종료 실패, `-1` = snapshot 실패.

#### B. `service_stop()` 끝에 `kill_process_by_fullpath` 호출 추가
- `service_command` 성공 후 같은 경로의 LMMAgent.exe 모두 정리.
- 모든 호출 경로 자동 적용 — 로그인 실패 (UdpSocket 의 5개 분기), 로그아웃, 재install 전.
- 종료 개수 > 0 일 때만 로그 출력.

#### C. `terminate_other_processes()` 의 `taskkill` 전면 교체
- 8개 ShellExecute(taskkill) → 8개 `kill_process_by_fullpath` 호출.
- 종료 대상은 그대로 유지하되 **현재 LMMLoginManager 의 설치 디렉토리 안의 exe 만** 종료.
- 옛 주석 코드에 누락되어 있던 nFTDClient2.exe / nFTDServer2.exe 추가.
- ShellExecute 비동기 → TerminateProcess 동기로 전환 — 함수 반환 시점에 종료 보장.

### 이전 vs 이후
| 시나리오 | 이전 | 이후 |
|---|---|---|
| 로그인 실패 후 잔재 | `-controlservice -slave` 살아남음 (반복 시 누적) | `service_stop()` 안에서 모두 정리 |
| `kill_process_by_fullpath` 호출 1회 | 1개만 종료 (호출자가 while 루프로 반복) | 모든 매칭 동시 종료 |
| `terminate_other_processes` + LMM 3.0 SE 공존 | SE 의 동명 프로세스도 종료 (부작용) | 본 제품 경로의 exe 만 종료 |
| 권한 부족 (PROCESS_ALL_ACCESS) | false negative — 종료 못 함 | 조회/종료 권한 분리로 성공률 ↑ |
| 함수 반환 시점의 종료 보장 | taskkill ShellExecute 비동기 → 없음 | TerminateProcess 동기 → 있음 |
